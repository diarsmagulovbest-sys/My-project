# PRODUCT_SPEC.md

## 1. Идея продукта

Приложение помогает пользователю достигать любых целей с помощью персонального AI-наставника.

Пользователь создаёт цель, например: «Я хочу научиться собирать кубик Рубика». После этого приложение должно:

1. Получить основную цель.
2. Узнать срок достижения цели.
3. Узнать, сколько времени пользователь может уделять цели каждый день или каждую неделю.
4. Задать персональные уточняющие вопросы.
5. Определить текущий уровень пользователя.
6. Создать персональную дорожную карту.
7. Разделить дорожную карту на этапы.
8. Создать конкретные задания.
9. Показывать сегодняшнее задание.
10. Позволять отмечать задания выполненными.
11. Сохранять прогресс.
12. Помогать пользователю, если он застрял.
13. Адаптировать план после пропусков, изменения срока или нехватки времени.
14. Работать как персональный AI-наставник.

Приложение должно подходить для разных целей: обучение, программирование, языки, хобби, творчество, спорт, личные проекты, подготовка к экзаменам и развитие навыков.

Не нужно делать отдельную жёсткую логику для каждой категории. AI должен создавать разные планы в зависимости от цели пользователя.

## 2. Главный пользовательский сценарий

1. Пользователь регистрируется или входит в аккаунт.
2. Пользователь нажимает «Создать цель».
3. Пользователь заполняет форму:
   - название или описание цели;
   - желаемый срок;
   - количество свободного времени;
   - единица времени: в день или в неделю;
   - дополнительные пожелания.
4. AI анализирует цель и создаёт от 5 до 6 уточняющих вопросов.
5. Пользователь отвечает на вопросы.
6. AI получает цель, срок, доступное время, ответы, текущий уровень и ограничения.
7. AI создаёт структурированный план.
8. Пользователь попадает на страницу цели.
9. Пользователь отмечает задания выполненными.
10. Приложение обновляет прогресс.
11. Если пользователь пропустил задания, он указывает причину, а AI предлагает изменение плана.

## 5. AI-провайдер

Сейчас используется Gemini через Supabase Edge Function `supabase/functions/ai`.

Текущая функция читает секрет `GEMINI_API_KEY` через `Deno.env.get('GEMINI_API_KEY')` и вызывает модель `gemini-2.0-flash`.

Не менять AI-провайдера на OpenAI или другой сервис без отдельного подтверждения.

Все AI-запросы должны идти через Supabase Edge Functions. Нельзя вызывать AI-модель напрямую из Vite-клиента.

Секретные AI-ключи:

- хранятся только в Supabase secrets или внутри окружения Edge Function;
- не попадают в клиентский код;
- не попадают в `.env` или `.env.local` с префиксом `VITE_`;
- не коммитятся в git.

Переменные `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` допустимы, потому что это публичные клиентские настройки Supabase. AI-ключи с префиксом `VITE_` запрещены.

## 7. База данных

База данных проектируется в Supabase через SQL-миграции.

Нужные сущности:

- `profiles`
- `goals`
- `goal_questions`
- `roadmap_stages`
- `tasks`
- `progress_logs`
- `conversations`
- `messages`

Основные поля:

```text
profiles:
  id, display_name, created_at, updated_at

goals:
  id, user_id, title, description, target_date, available_time,
  time_period, current_level, status, progress, created_at, updated_at

goal_questions:
  id, goal_id, question, answer, sort_order, created_at

roadmap_stages:
  id, goal_id, title, description, success_criteria, sort_order, status, created_at

tasks:
  id, goal_id, stage_id, title, description, estimated_minutes,
  due_date, sort_order, status, completed_at, created_at, updated_at

progress_logs:
  id, goal_id, task_id, user_id, action, note, created_at

conversations:
  id, goal_id, user_id, created_at, updated_at

messages:
  id, conversation_id, role, content, created_at
```

Row Level Security обязателен.

Пользователь должен иметь доступ только к собственным:

- профилю;
- целям;
- вопросам;
- этапам;
- заданиям;
- журналу прогресса;
- разговорам;
- сообщениям.

Нельзя полагаться только на фильтрацию в интерфейсе. Ограничение доступа должно работать на уровне базы данных.

## 8. AI-логика

Нужны три AI-сценария через Supabase Edge Function.

### Генерация уточняющих вопросов

Вход:

- цель;
- описание;
- срок;
- доступное время;
- дополнительные пожелания.

Результат:

- от 5 до 6 коротких вопросов;
- вопросы помогают сделать план точнее;
- вопросы не повторяют уже известную информацию.

### Генерация дорожной карты

Вход:

- цель;
- срок;
- доступное время;
- ответы на уточняющие вопросы.

Результат должен иметь строгую JSON-структуру:

```json
{
  "goalSummary": "string",
  "feasibility": {
    "status": "realistic | challenging | unrealistic",
    "explanation": "string"
  },
  "mentorAdvice": "string",
  "stages": [
    {
      "title": "string",
      "description": "string",
      "successCriteria": ["string"],
      "tasks": [
        {
          "title": "string",
          "description": "string",
          "estimatedMinutes": 30,
          "suggestedDayOffset": 0
        }
      ]
    }
  ]
}
```

Для проверки структурированных AI-ответов рекомендуется добавить Zod отдельным маленьким этапом. До добавления Zod не сохранять AI-ответ без ручной проверки формы данных.

Если AI-ответ не соответствует схеме:

- не сохранять повреждённые данные;
- показать понятную ошибку;
- предложить повторить генерацию.

### AI-наставник

AI-наставник должен знать релевантный контекст цели:

- цель;
- срок;
- доступное время;
- ответы;
- дорожную карту;
- выполненные и пропущенные задания;
- текущий прогресс.

AI-наставник должен:

- отвечать по теме цели;
- объяснять непонятные задания;
- помогать разбивать сложные задачи;
- предлагать реалистичные следующие действия;
- не менять сохранённый план без подтверждения пользователя.

Стиль:

- современный;
- минималистичный;
- спокойный;
- профессиональный;
- без визуального шума;
- без большого количества градиентов;
- без случайных декоративных элементов.

Основные экраны:

1. Регистрация.
2. Вход.

4. Страница создания цели.
5. Страница уточняющих вопросов.
6. Экран генерации плана.
7. Страница одной цели.
8. Панель AI-наставника.

На странице цели должны быть:

- заголовок;

- текущее задание;
- дорожная карта;
- этапы;
- список заданий;

## 10. Качество кода

Требования:

- TypeScript без необоснованного `any`;
- понятные названия переменных;
- небольшие компоненты;
- отсутствие дублирования;
- валидация пользовательского ввода;
- обработка ошибок;
- безопасная работа с API-ключами;
- отсутствие мёртвого кода;
- отсутствие временных заглушек в завершённых функциях.

Не добавлять зависимости без причины.

ESLint сейчас не настроен. Его нужно добавить отдельным маленьким этапом перед следующими большими фичами.

## 11. Проверки

После каждого этапа:

1. Запустить доступный lint, если ESLint уже настроен.
2. Запустить TypeScript-проверку.
3. Запустить production build.
4. Исправить ошибки, появившиеся из-за изменений.

Текущие команды:

- `npm run build` — TypeScript-проверка + Vite production build.
- `npx tsc -b` — отдельная TypeScript-проверка.

После настройки ESLint:

- `npm run lint`.
